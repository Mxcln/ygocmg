# P3.5 Summary

## 1. Scope

This update implements the P3.5 "unified confirmation and warning flow" as a card-focused vertical slice.

Included in this slice:

- Shared in-app confirmation / warning dialog
- Replace browser-native confirm for `Delete Pack`
- Replace browser-native confirm for `Delete Card`
- Real staged confirmation for `create_card` / `update_card`
- Backend confirmation token cache and `confirm_card_write`
- `delete_card` migrated to `WriteResult` contract

Not included in this slice:

- Unsaved-close interception
- Strings write confirmation
- Resource write confirmation
- Import/export preview-token flow
- Batch card operations

## 2. Frontend Changes

### 2.1 Shared App Dialog

A shared `AppDialog` was added at the app shell level:

- Path: `src/features/dialogs/AppDialog.tsx`
- Mounted from: `src/app/App.tsx`

Supported dialog modes:

- `confirm`
  - Used for destructive or blocking actions
  - Supports title, summary, confirm/cancel labels, danger styling, async confirm
- `warning`
  - Used for `needs_confirmation` flows
  - Supports title, summary, warning list, continue/cancel actions

Behavior:

- Runs above the main shell and card drawer
- Uses app-defined UI instead of browser-native `window.confirm`
- Locks closing while async confirmation is running
- Catches async confirm failures and keeps the error visible inside the dialog
- Supports backdrop click to close when not busy
- Supports `Escape` to cancel and `Enter` to confirm
- After follow-up review, the redundant top-right `Close` button was removed

### 2.2 Delete Pack / Delete Card

`Delete Pack` and `Delete Card` now both use the shared app dialog.

- `Delete Pack`
  - Triggered from pack metadata actions in `src/app/App.tsx`
  - Confirmed in dialog, then calls existing delete-pack backend flow

- `Delete Card`
  - Triggered from `src/features/card/CardEditDrawer.tsx`
  - Confirmed in dialog, then calls `cardApi.deleteCard(...)`
  - Frontend expects `WriteResult.status === "ok"`
  - If backend ever returns `needs_confirmation` for delete in the future, current frontend treats that as unsupported rather than silently continuing

### 2.3 Card Save Warning Flow

`CardEditDrawer` now handles staged confirmation:

- First call `createCard` or `updateCard`
- If result is `ok`, save completes immediately
- If result is `needs_confirmation`, open warning dialog
- User chooses:
  - `Cancel`: do not persist
  - `Continue`: call `confirmCardWrite({ confirmationToken })`

The confirmed write returns final `CardDetail`, which is then used to refresh the drawer state and close it through the normal saved flow.

If confirmation fails, the failure now stays visible in the dialog layer instead of being lost behind the drawer.

## 3. Backend Changes

### 3.1 Confirmation Cache

A runtime confirmation cache was added:

- Path: `src-tauri/src/runtime/confirmation_cache.rs`
- Stored in: `AppState.confirmation_cache`

Each staged card write stores:

- `confirmation_token`
- `workspace_id`
- `pack_id`
- `pack_revision`
- `source_stamp`
- `operation_kind`
  - `create_card`
  - `update_card`
- `input_snapshot`
- `create_card_seed` for create flows
- `warnings`

Current token semantics:

- Single-use
- Removed after successful confirm
- Invalidated when pack/workspace state changes
- Rejected if `revision` or `source_stamp` no longer match

No time-based TTL was added in this slice.

### 3.2 CardWriteConfirmationService

A dedicated application service was added:

- Path: `src-tauri/src/application/card/confirmation_service.rs`

Responsibilities:

- Run create/update preflight
- Decide between immediate write vs staged confirmation
- Persist staged confirmation entry
- Confirm token and execute final commit

### 3.3 Create / Update Split into Prepare + Commit

`PackWriteService` was refactored so create/update no longer only have a direct write path.

The flow is now:

1. Normalize input
2. Validate structure and code policy
3. Collect warnings
4. If errors exist: return error
5. If warnings exist and not yet confirmed:
   - stage entry in confirmation cache
   - return `WriteResultDto::NeedsConfirmation`
6. Otherwise:
   - commit write
   - replace session
   - refresh summaries
   - return `WriteResultDto::Ok`

This commit path is shared by:

- immediate create/update without warnings
- confirmed create/update through `confirm_card_write`

### 3.4 confirm_card_write

New command added:

- Rust command: `confirm_card_write`
- TS API: `cardApi.confirmCardWrite(...)`

Behavior:

1. Consume token from confirmation cache
2. Re-check current open-pack snapshot
3. Validate `revision` and `source_stamp`
4. Rebuild write context
5. Re-run prepared write
   - create confirmation reuses the staged card seed so final `id` / `created_at` stay stable
6. Commit and return final `CardDetailDto`

If validation fails, backend returns a confirmation-related error such as:

- `confirmation.invalid_token`
- `confirmation.stale_revision`
- `confirmation.stale_source_stamp`

### 3.5 delete_card Contract Update

`delete_card` was updated to use the shared write-result contract.

Contract changes:

- Input now requires explicit `workspaceId`
- Return type is now `WriteResult<DeleteCardResult>`
- `DeleteCardResult = { deleted_card_id: string }`

Current behavior:

- Delete is still direct
- It does not use staged warning confirmation in this slice
- Successful delete returns `status: "ok"` and `warnings: []`

## 4. State Invalidation Rules

Pending confirmation tokens are invalidated on relevant state changes.

Current invalidation points include:

- pack write completion through session replacement
- pack reopen / open
- pack close
- pack delete
- workspace switch / open
- workspace clear

This keeps staged tokens tied to a specific pack snapshot and prevents stale confirmation replay.

## 5. UX Message Channels

One important outcome of this work is that the app now clearly uses multiple feedback channels with different responsibilities.

### 5.1 Shell Notice

The notice bar above the pack metadata area is a shell-level message channel.

- State lives in: `src/app/App.tsx`
- Rendered near the top of the `work-area`
- Used by:
  - workspace open/create
  - settings save
  - pack open/create/delete
  - pack metadata save
  - other shell-level failures

Why this pattern is used:

- These events affect workspace or pack shell state
- The initiating modal may already be closed when feedback is shown
- A top-level notice keeps the result visible across modal boundaries

### 5.2 Drawer-local Error / Warning

Card editing uses drawer-local feedback for form-specific failures.

- State lives in: `src/features/card/CardEditDrawer.tsx`
- Used for:
  - invalid save attempts
  - backend validation errors
  - card-specific failures

This means card validation failures are intended to stay near the card form rather than being pushed into the shell notice area.

### 5.3 Confirmation / Warning Dialog

The new dialog layer is for user decisions, not passive status reporting.

- `confirm`: destructive approval
- `warning`: continue-or-cancel decision for `needs_confirmation`

So the three channels now have distinct jobs:

- Notice: shell-level outcome/status
- Drawer error: local card-edit problem
- Dialog: blocking decision point

## 6. Contracts Added or Updated

Backend / frontend contracts added or changed in this slice:

- `ConfirmCardWriteInput`
- `DeleteCardInput`
  - now includes `workspaceId`
- `DeleteCardResult`
- `WriteResultDto::NeedsConfirmation.confirmation_token`
  - now uses a dedicated token alias on Rust side
- `cardApi.confirmCardWrite(...)`
- `cardApi.deleteCard(...)`
  - now returns `WriteResult<DeleteCardResult>`

## 7. Validation

Validated during implementation with:

- `npm run typecheck`
- `cargo test --offline`

Rust coverage added/updated for:

- create warning -> confirm -> persist
- update warning -> confirm -> persist
- token single-use behavior
- stale token invalidation on pack revision change
- create confirmation reusing staged card identity
- `delete_card` returning `WriteResult::Ok`
- `workspace_id` mismatch rejection

## 8. Known Limits

Current limits of this slice:

- Warning dialog now has a small code-to-text mapping for common card-code warnings
  - there is still no full i18n / centralized warning-copy layer yet
- `delete_card` is unified at the contract layer but still not a staged warning flow
- unsaved-close confirmation is still not wired into this dialog system
- strings / resource / import-export flows are still outside this implementation

## 9. Review Focus

Recommended review focus areas:

1. Does the split between shell notice, drawer-local error, and decision dialog feel correct?
2. Is token invalidation strict enough for pack revision and source changes?
3. Is the current warning presentation acceptable until code-to-copy mapping is added?
4. Should backdrop click continue to behave like `Cancel`, or should warning dialogs become stricter?
5. When this expands beyond cards, should the confirmation cache stay card-specific or be generalized immediately?
