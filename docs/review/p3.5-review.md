### 一、架构与设计（整体好评）

1. **Prepare/Commit 拆分很干净**: `PackWriteService` 从原来的单体 `create_card` / `update_card` 拆成 `prepare_*` + `commit_*` 两阶段，使确认流可以插入而不复制逻辑。抽取的 `replace_and_refresh` 消除了三处重复的 session 替换 + summary 刷新代码。

2. **`CardWriteConfirmationService` 职责清晰**: 作为表示层和写入服务之间的编排层，承担了"预检 → 决定是否暂存 → 确认时重放"的完整流程，没有把确认逻辑散落到 presentation 层或 write service 内部。

3. **ConfirmationCache 简洁可控**: 基于 `BTreeMap<Token, Entry>`，提供 pack 级、workspace 级和全量清除三种失效粒度。失效点覆盖了 pack write / pack close / pack delete / pack open / workspace switch / workspace clear，覆盖面充分。

4. **三通道 UX 模型清晰**: Shell notice（全局操作结果）、Drawer-local error（卡片表单错误）、Dialog（阻塞式决策点）三层分工合理，不会相互覆盖职责。

5. **测试覆盖到位**: 新增的 `confirmation_token_can_only_be_used_once_and_expires_on_revision_change` 和 `delete_card_returns_write_result_and_rejects_workspace_mismatch` 覆盖了核心不变量（单次使用、revision 变化后失效、workspace_id 不匹配拒绝）。

### 二、需要关注的问题

#### 高优先级

**1. `handleConfirm` 异常处理存在 UX 死角**

`AppDialog.tsx` 的 `handleConfirm` 只有 `try/finally` 没有 `catch`：

```12:18:src/features/dialogs/AppDialog.tsx
  async function handleConfirm() {
    try {
      setDialogBusy(true);
      await dialog.onConfirm();
    } finally {
      setDialogBusy(false);
    }
  }
```

而 `CardEditDrawer.tsx` 的 `onConfirm` 回调在出错时会 `throw err`：

```typescript
// confirmCardWrite 的 onConfirm 中
} catch (err) {
  setErrorMsg(formatError(err));
  throw err;  // 抛到 handleConfirm，变成 unhandled promise rejection
}
```

当确认失败时：
- 错误被 `setErrorMsg` 设置到了 **drawer 上**，但 dialog 仍然覆盖在 drawer 之上（z-index 140 > 20）
- 用户看到 dialog 重置为非 busy 状态，但看不到底下的错误信息
- `throw err` 变成了未捕获的 Promise rejection

**建议**: `handleConfirm` 应该 catch 异常后把错误显示在 dialog 内部，或者自动关闭 dialog 让 drawer 的错误可见。

**2. `confirm_card_write` 对 create 场景会重新生成 UUID**

`confirm_card_write` 在确认时重新调用 `prepare_create_card`，而 `prepare_create_card` 每次都会调用 `Uuid::now_v7()` 生成新的 card ID。这意味着暂存时预览的 card 信息和最终写入的 card 信息会有不同的 `id` 和 `created_at` 时间戳。

虽然当前前端没有在 NeedsConfirmation 阶段展示 card ID，所以实际无害，但如果未来 `preview` 字段开始返回预览卡片，这个不一致会浮出来。

**建议**: 考虑在 `CardConfirmationInputSnapshot` 中存储首次生成的 card ID，在确认时重用。

#### 中优先级

**3. 缺少键盘交互**

`AppDialog` 没有处理 `Escape` 键关闭和 `Enter` 键确认。作为模态对话框的标准无障碍要求，这在 P3.5 范围内值得补上。

**4. `get_card_entry` 是死代码**

`ConfirmationCache` 上的 `get_card_entry` 方法已定义但全项目没有任何调用者：

```49:51:src-tauri/src/runtime/confirmation_cache.rs
    pub fn get_card_entry(&self, token: &ConfirmationToken) -> Option<&CardConfirmationEntry> {
        self.card_entries.get(token)
    }
```

**建议**: 删除，或加 `#[allow(dead_code)]` 并注释说明预留用途。

**5. `commit_prepared_update_card_session` 存在冗余的 post-validation**

`prepare_update_card` 已经对 `updated` 做了 `validate_card_structure` 检查并在有 error 时返回错误。但 `commit_prepared_update_card_session` 在将 `prepared.updated` 放入 `next_cards` 后又做了一次完全相同的检查。两次检查的输入数据完全相同（`*card = prepared.updated.clone()` 后立即 `validate_card_structure(card)`），所以第二次不会捕获任何新问题。

这个冗余是从 P3 继承的，不是 P3.5 引入的，可以在后续清理。

**6. `commit_prepared_create_card_session` 重新获取 `now_utc()` 导致时间戳分裂**

```typescript
// prepare 阶段: card.created_at = now_utc() (时间 T1)
// commit 阶段: metadata.updated_at = now_utc() (时间 T2)
```

对于确认流（prepare 和 commit 之间可能隔几秒到几分钟），card 的创建时间和 metadata 的更新时间会明显不同。虽然语义上可以接受（metadata 应该记录实际写入时间），但值得在文档中明确这是有意为之。

#### 低优先级

**7. Warning 展示目前是原始 code + JSON.stringify(params)**

Summary 已标记为 Known Limits。当前用户看到的是类似 `code.outside_recommended_range {"min":100000000,"max":200000000}` 的技术性文本，可用但不友好。建议在 P4 或更早时加入 code → 人类可读文本的映射。

**8. `preview` 字段始终为 `None`/`null`**

`WriteResultDto::NeedsConfirmation` 的 `preview` 字段当前从未被填充。前端合约中也已改为 `unknown | null`。如果短期内不打算使用，可以考虑是否需要保留在合约中。不过保留它对前向兼容性有益，不是问题。

**9. Token 无时间 TTL**

Summary 已标记。Tokens 只有 revision/stamp 失效和主动失效，没有时间过期。用户可以把 warning dialog 放置几小时后再确认。由于 revision/stamp 检查已经保护了状态一致性，这在当前阶段可以接受。
